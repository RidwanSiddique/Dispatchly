Dispatchly — Strengths vs. Gaps

An honest audit of where Dispatchly competes well and where the big platforms pull ahead

✅ What Dispatchly Does Well
Full ITSM core triangle — Incidents, SLA monitoring, and escalation are solid. Matches what most teams use day-to-day in Jira/Zendesk.
ITIL-aligned Problem & Change Management — Both modules are more complete than Zendesk's out-of-the-box ITSM, which doesn't natively support Problem/Change.
Role-based visibility done right — 7 granular roles with scoped ticket visibility, requester self-service isolation, and technician accept/decline flow. More nuanced than basic Zendesk RBAC.
SLA auto-escalation — Background cron that auto-escalates on breach with dedup via sla_alerts. Feature parity with Jira SM on this specific flow.
P1 on-call auto-assignment — Automatically pages and assigns the on-call tech when a P1 ticket lands. Matches core PagerDuty-style paging logic.
Ticket → KB conversion — Resolved tickets can be published directly as KB articles. Feature present in all major platforms but often overlooked in smaller tools.
Problem → bulk ticket resolution — When a problem is resolved, all linked tickets auto-resolve and their requesters are notified. Jira SM does this; Zendesk doesn't natively.
Service catalog with approval flows — Catalog items can require approval before creating a ticket, with status "Pending Approval". On par with Jira SM's basic catalog.
Clean, custom codebase — No licensing fees, no per-seat costs, full control of data and customization. This is the core competitive advantage vs. $20k+/yr ServiceNow.
❌ Critical Gaps vs. Enterprise Platforms
No AI / ML anywhere — No auto-triage, no suggested KB articles during ticket creation, no generative summaries. All three competitors ship AI as table stakes in 2026.
No email notifications — Notifications exist only inside the app bell. No outbound emails for status updates, assignments, or SLA alerts — a major gap for any real helpdesk.
No asset management / CMDB — Can't track IT assets or link them to tickets. ServiceNow and Jira SM treat this as a first-class module. Critical for enterprise IT.
No business-hours SLA — SLA clock runs 24/7 including weekends. All competitors support business-hour schedules (SLA pauses off-hours). This inflates breach rates unfairly.
No workflow automation builder — Every trigger (notifications, status transitions) is hardcoded. ServiceNow Flow Designer, Jira Automation, and Zendesk Action Builder let admins build rules without code.
No SSO / OAuth / SAML — Email + password only. Enterprise IT teams require SSO (Okta, Azure AD, Google Workspace). Blocks adoption in mid/large orgs.
No audit trail — No timestamped log of who changed what. Compliance-heavy industries (healthcare, finance) legally require this.
No advanced analytics — Only a basic dashboard. No agent performance tracking, no exportable reports, no SLA compliance trends over time.
Skills not used for routing — Skills are stored on users but never used for auto-routing or availability matching. Jira SM and ServiceNow use skills + availability to pick the right agent.
No mobile app or omnichannel — No native mobile, no chat widget, no phone/SMS integration. Real helpdesks need agents on mobile for on-call responses.
Time tracking not surfaced in UI — The time_entries table exists in the DB (migration 015) but there's no frontend page or workflow built around it.
⚡ Where Dispatchly Unexpectedly Beats Zendesk
Full Problem Management (root cause, workaround, bulk resolve) — Zendesk doesn't have this natively
Full Change Management with approval workflow — Zendesk has no change module
Technician accept/decline flow — Zendesk has no such handshake
P1 auto-assign to on-call — Zendesk requires third-party paging tools
Zero cost — Zendesk charges ~$55–$115/agent/mo

What to Build Next

Prioritized features to close the gap with enterprise platforms — ranked by impact vs. effort

🔥 High Impact · Low Effort (Build First)

📧 Email Notifications
Critical gap — currently notifications are in-app only
Easy
Use Nodemailer or SendGrid. Hook into existing notifyUser() — just add an email send alongside the DB insert. ~1–2 days work.
⏰ Business-Hours SLA
SLA clock runs 24/7 — unfair to agents on nights/weekends
Easy
Modify computeSla() to subtract non-business minutes. Add a schedule config (business hours per day). ~1 day.
📈 Time Tracking UI
DB table (time_entries) already exists — just needs a frontend
Easy
Add a "Log Time" button on TicketDetailPage. Show total time per ticket on the detail view. Routes and DB already there. ~1 day.
🛠 Audit Log
Who changed what and when — required for compliance
Easy
New audit_log table. Hook into PATCH /tickets/:id, updateProblem, updateChange to INSERT before/after snapshots. ~1–2 days.
🔍 Skills-Based Auto-Routing
Skills already stored — route tickets to best-matched agent
Easy
Match ticket category → agent skills array. Auto-suggest assignment when a ticket is created or escalated. Pure backend logic, ~1 day.
📊 SLA Compliance Report
Time-based SLA trend — which priority breaches most?
Easy
Extend /api/dashboard with a 30/60/90-day SLA compliance breakdown by priority. Add a chart in DashboardPage. ~1 day.

🚀 High Impact · Medium Effort (Build Next)

🤖 AI Ticket Triage
Auto-classify category + priority from ticket title/body
Medium
Call OpenAI/Claude API on ticket creation. Suggest category, priority, and a matching KB article. Show suggestions — agent confirms. ~3–5 days.
📱 Custom Notification Rules
Let admins configure who gets notified for what events
Medium
New notification_rules table. UI in admin panel. Replace hardcoded recipient logic in controllers with rule lookups. ~3–5 days.
📋 Change Calendar View
Visual calendar of upcoming maintenance windows
Medium
Add a calendar component (FullCalendar.js) to ChangesPage. Plot maintenance_window_start/end for approved changes. ~2–3 days.
🔐 SSO Integration
Google / Microsoft / Okta sign-in for enterprise orgs
Medium
Add Passport.js with Google OAuth / SAML 2.0 strategy. Issue same JWT on success. Blocks enterprise adoption without this. ~3–5 days.
📊 Agent Performance Dashboard
Tickets closed, avg resolution time, SLA score per agent
Medium
New /api/reports/agents endpoint. Aggregate from tickets table grouped by assigned_to_user_id. Add manager-only view. ~2–3 days.
🗄 Asset Management (Basic)
Track IT assets and link them to tickets
Medium
New assets table (name, type, serial, owner, status). Link to tickets via ticket_assets. New /assets routes + AssetsPage. ~5–7 days.

🏔 Game-Changers · High Effort

🤖 AI Virtual Agent / Chatbot
Self-service portal with AI that deflects common requests
Hard
Chat widget powered by LLM + KB retrieval (RAG). Deflects ~30–40% of tickets before they're created. Major differentiator if built well.
📱 Mobile App
React Native app for agents responding on-call
Hard
Critical for on-call workflows. Push notifications for P1 alerts. Accept/decline assignment from phone. Backend API is already mobile-ready.
🔗 Slack / Teams Integration
Create, update, and get notified on tickets via Slack
Hard
Slack bot that posts P1/SLA alerts to channels. Allow ticket actions via Slack slash commands. Major workflow improvement for DevOps teams.

Market Positioning

Where each platform is designed to live, and where Dispatchly fits in that landscape

🟢 ServiceNow ITSM
Target: Large enterprises (1,000+ employees), complex cross-functional IT orgs
Strengths: Most comprehensive platform on earth. AI-first (post-Moveworks acquisition). CMDB as the single source of truth. Unmatched workflow automation.
Weaknesses: Extremely expensive ($10k–$20k+/yr, scaling 3x with growth). Months-long implementation. Requires dedicated ServiceNow admins/developers.
Pricing: Custom quotes — no public pricing. Foundation → Advanced → Prime tiers. AI add-ons sold separately.
🔵 Jira Service Management
Target: SMBs to mid-market with engineering/DevOps teams already on Atlassian ecosystem
Strengths: Best value for dev-centric ITSM. Deep Jira/Confluence/Bitbucket integration. 8,000+ marketplace apps. CI/CD-native change management.
Weaknesses: Problem/Change/Major Incident moved to Premium plan (Oct 2024). Can get expensive at scale. UI complexity intimidates non-technical users.
Pricing: ~$20/agent/mo (Standard) · ~$45–$60/agent/mo (Premium). Free tier: up to 3 agents.
🟡 Zendesk
Target: Customer-facing support teams (CX-first). SMB to mid-market. Less ITSM depth.
Strengths: Best omnichannel (email, chat, voice, social). Strong AI Copilot for agents. Fastest time-to-value. Great for customer service orgs.
Weaknesses: Lacks native ITIL Problem/Change Management. CMDB = essentially absent. Not ideal for internal IT helpdesk teams wanting ITSM depth.
Pricing: ~$55/agent/mo (Suite Team) · ~$115/agent/mo (Suite Professional)
⚡ Dispatchly (Ours)
Target: Small-to-mid IT teams, healthcare/clinical environments, organizations needing custom ITSM without SaaS lock-in or per-seat costs
Unique strengths: Zero licensing cost. Full data ownership. Custom-built role model (7 roles, REQUESTER vs STAFF split). Complete Incident + Problem + Change + Catalog in one codebase.
Sweet spot: Orgs that can't afford ServiceNow, don't want Jira's Atlassian ecosystem lock-in, and need more ITSM depth than Zendesk provides — but have dev resources to run and extend the system.
Pricing: Self-hosted = hosting cost only. No per-seat fees. At 20 agents, you save $10,000–$20,000/yr vs. Jira SM Premium, and $100k+/yr vs. ServiceNow.
⚡ Dispatchly's Positioning in One Sentence
Dispatchly is an open, self-hosted ITSM platform that covers the full ITIL core (Incident → Problem → Change → Catalog → KB) with zero licensing cost, making it the right choice for organizations that want enterprise-grade workflows without enterprise pricing — and have the engineering capacity to own their tooling.
The gap to close in 2026: Adding email notifications, business-hours SLA, SSO, and basic AI triage would push Dispatchly from "solid internal tool" to "credible alternative to Jira SM Standard" for teams under 100 agents. These are all 1–5 day engineering tasks — the architecture already supports them.
